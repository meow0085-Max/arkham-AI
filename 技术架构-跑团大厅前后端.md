# Arkham AI 跑团大厅 — 前后端技术架构说明

## 一、目标架构（你期望的形态）

- **跑团大厅**：只做**前端页面**，负责与用户交互（展示界面、收集输入、显示结果）。
- **用户输入**：通过**调用后端 API** 发送到服务器。
- **后端**：接收请求、调用 AI（如 Gemini）、处理模组解析等，把**结果返回给前端**。
- **前端**：收到后端返回后，在跑团大厅中展示（例如守秘人回复、模组解析结果）。

即：**前端 ↔ HTTP API ↔ 后端**，AI 与敏感配置（如 API Key）放在后端，前端不直连第三方 AI。

---

## 二、当前实现 vs 目标架构

| 能力           | 当前实现                         | 目标架构                           |
|----------------|----------------------------------|------------------------------------|
| 角色卡同步     | 前端请求 `POST/GET /api/investigator`（后端已有） | 保持不变，仍由后端提供             |
| 守秘人对话     | 前端用 localStorage 的 Key **直连** Gemini API | 改为：前端只发「用户消息+上下文」到后端，**后端**调 Gemini，再返回回复 |
| 模组 PDF 解析  | 前端把 PDF 转 base64 **直连** Gemini           | 改为：前端上传 PDF 或 base64 到**后端**，后端调 Gemini 解析，返回模组大纲文本 |
| API Key        | 存在浏览器 localStorage，暴露在客户端         | 放在**后端**环境变量/配置，前端不再传 Key |

## 三、整体数据流（目标架构）

```
┌─────────────────────────────────────────────────────────────────┐
│                        前端（跑团大厅）                           │
│  - 展示：聊天记录、模组预览、调查员信息、API Key 配置区（可改为“连接后端”）  │
│  - 收集：用户输入、当前调查员、当前模组文本、对话历史、上传的 PDF          │
│  - 只请求自己的后端 API，不直连 Gemini                             │
└───────────────────────────┬─────────────────────────────────────┘
                            │ HTTP（JSON / FormData）
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                    后端（如 Node + Express）                       │
│  - 接收前端请求                                                   │
│  - 使用服务器上的 Gemini API Key 调用 Google API                   │
│  - 守秘人对话：组 systemInstruction + 对话历史 → Gemini → 回复     │
│  - 模组解析：收 PDF（或 base64）→ Gemini（inlineData）→ 模组大纲   │
│  - 将结果以 JSON 等形式返回给前端                                  │
└───────────────────────────┬─────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                    外部服务（如 Google Gemini）                     │
└─────────────────────────────────────────────────────────────────┘
```

## 四、前端（跑团大厅）职责

1. **页面与交互**
   - 跑团大厅视图的展示（聊天区、模组上传区、调查员选择、设置区等）。
   - 用户输入（打字、发送）、上传文件（.txt/.md/.json/PDF）的收集。

2. **状态管理（仅前端状态）**
   - 当前选中的调查员、当前模组文本（或模组大纲）、当前对话历史。
   - 这些在「调用后端」时作为请求体的一部分发给后端，而不是由后端持久化会话（除非你后续要做「会话存库」）。

3. **只调用后端 API**
   - **守秘人回复**：把「用户本条消息 + 当前调查员 + 当前模组上下文 + 对话历史」发给后端某个接口（如 `POST /api/keeper/chat`），拿回「守秘人回复」并追加到聊天记录。
   - **模组解析（PDF）**：把 PDF 文件（或 base64）发给后端某个接口（如 `POST /api/keeper/parse-module`），拿回「模组大纲文本」，前端写入当前模组上下文并更新模组预览。
   - **.txt/.md/.json 模组**：可以继续在前端用 FileReader 读成文本，只把「模组文本」存在前端内存用于后续发给 `/api/keeper/chat`；若你希望「模组也经后端处理」，再增加一个「解析文本模组」的接口即可。

4. **不再做的事**
   - 不直连 `generativelanguage.googleapis.com`。
   - 不在前端保存或上传 Gemini API Key；可改为「后端地址」或「是否已连接后端」的配置。

---

## 五、后端职责

1. **提供 HTTP API**
   - **守秘人对话**：接收前端发来的「角色卡 + 模组上下文 + 对话历史 + 用户本条消息」，用后端配置的 Gemini Key 调 `generateContent`。**必须将角色卡完整传入 AI 的上下文**（如 systemInstruction），使守秘人能根据属性与技能进行数值检定（见 6.1.1）；再把生成的守秘人回复返回给前端。
   - **模组 PDF 解析**：接收 PDF（multipart 或 JSON 内 base64），用 Gemini 的 inlineData 调用一次「提取/总结模组」，把得到的模组大纲文本返回给前端。

2. **安全与配置**
   - Gemini API Key 放在后端环境变量（或配置文件），不暴露给浏览器。
   - 可选：限流、鉴权（如简单 Token）、请求体大小限制（尤其是 PDF）。

3. **可选扩展**
   - 角色卡：你已有 `POST/GET /api/investigator`，可保持不动。
   - 若将来要做「会话存档」，可在后端为每次跑团建会话、存对话与模组摘要。

## 六、API 接口详细设计

以下为跑团大厅前后端对接的完整接口约定，实现时请严格按字段名与类型使用。

---

### 6.1 守秘人对话

获取 AI 守秘人对「用户本条消息」的回复，需携带当前调查员、模组上下文与历史对话。**后端必须将角色卡完整传给 AI**，以便守秘人根据属性与技能进行数值检定（见 6.1.1）。

| 项目 | 说明 |
|------|------|
| **方法** | `POST` |
| **路径** | `/api/keeper/chat` |
| **Content-Type** | `application/json` |

**请求体（Request Body）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `character` | `object` | 是 | 当前调查员角色卡，与前端「建立调查员」导出的 JSON 一致（含姓名、属性、技能等）。**AI 需据此进行属性/技能检定**，见 6.1.1。可为 `{}` 表示无角色卡。 |
| `moduleContext` | `string` | 是 | 当前模组全文或大纲文本，用于约束守秘人叙事。可为空字符串 `""`。 |
| `chatHistory` | `array` | 是 | 本轮跑团的历史消息列表，按时间顺序。每项见下表。 |
| `userMessage` | `string` | 是 | 用户**本条**输入（即本次要守秘人回复的那一条）。 |

**chatHistory 单项结构**

| 字段 | 类型 | 说明 |
|------|------|------|
| `role` | `string` | 固定为 `"user"` 或 `"assistant"`，表示发言方。 |
| `content` | `string` | 该条消息的完整文本。 |

**请求示例**

```json
{
  "character": { "name": "张三", "str": 55, "con": 60, "skills": {} },
  "moduleContext": "模组：黑暗中的低语。场景一：调查员抵达老宅……",
  "chatHistory": [
    { "role": "assistant", "content": "空气骤然凝滞……" },
    { "role": "user", "content": "我推开门看看" }
  ],
  "userMessage": "我推开门看看"
}
```

**响应（Response）**

- **成功**：HTTP 200，JSON 体

| 字段 | 类型 | 说明 |
|------|------|------|
| `ok` | `boolean` | 固定 `true`。 |
| `text` | `string` | 守秘人回复的完整文本，前端直接追加到聊天记录并展示。 |

示例：`{ "ok": true, "text": "你缓缓推开那扇吱呀作响的木门……" }`

- **失败**：HTTP 4xx/5xx 或 200 但 `ok: false`，JSON 体

| 字段 | 类型 | 说明 |
|------|------|------|
| `ok` | `boolean` | 固定 `false`。 |
| `error` | `string` | 错误信息，用于前端提示（如「API Key 未配置」「Gemini 调用超时」）。 |

示例：`{ "ok": false, "error": "Gemini 调用失败：API key invalid" }`

**6.1.1 角色卡与数值检定**

守秘人需要根据用户建立的角色卡进行 CoC 7th 的**数值检定**（属性检定、技能检定）。前端在每次请求 `POST /api/keeper/chat` 时都会带上当前选中的 `character`；后端在调用 AI 时，必须把该角色卡**完整放入发给模型的上下文**（如 systemInstruction 或等效内容），使 AI 能读取并依据下列数据判定检定与叙事：

- **基础属性**：用于属性检定（如力量、敏捷、体质等）。结构为 `attributes[属性名]`，每项包含：
  - `value`：常规难度出目目标（如 STR 55 → 骰出 ≤55 即成功）
  - `half`：困难难度（≤ 一半）
  - `fifth`：极难难度（≤ 五分之一）
- **技能**：用于技能检定（如侦查、图书馆利用、潜行等）。结构为 `skills[技能名]`，每项包含 `occ`（职业点）与 `interest`（兴趣点），**检定目标值 = occ + interest**（百分比）。
- **其他**：`name`、`occupation`、`age`、`gender`、`background` 等供叙事与身份引用。

后端在构造给 AI 的提示时，应明确要求守秘人：根据角色卡中的属性与技能数值进行检定；需要掷骰时，明确写出「该技能/属性为 XX，需骰出 ≤XX 才成功」等，以便玩家按规则执行。角色卡 JSON 建议原样或格式化后放入「当前调查员状态」等段落，便于模型直接读取数值。

**角色卡结构示例（与「建立调查员」页导出一致）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | `string` | 调查员姓名 |
| `occupation` | `string` | 职业 |
| `age` | `number` | 年龄 |
| `gender` | `string` | 性别 |
| `attributes` | `object` | 属性表，键为 STR/CON/SIZ/DEX/INT/APP/POW/EDU 等，值为 `{ value, half, fifth }` |
| `skills` | `object` | 技能表，键为技能名（如「侦查」「图书馆利用」），值为 `{ occ, interest }`，检定值 = occ + interest |
| `background` | `string` | 背景描述（可选） |

### 6.2 模组 PDF 解析（AI 解析）

上传 PDF 模组，由后端调用 Gemini 提取/总结为模组大纲文本，供后续守秘人对话使用。

| 项目 | 说明 |
|------|------|
| **方法** | `POST` |
| **路径** | `/api/keeper/parse-module` |
| **请求方式** | 二选一：**A. JSON（base64）** 或 **B. multipart/form-data（文件）** |

**方式 A：JSON + base64**

| Header | 值 |
|--------|-----|
| Content-Type | `application/json` |

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `pdfBase64` | `string` | 是 | PDF 文件的 Base64 编码（不含 `data:application/pdf;base64,` 前缀）。 |

示例：`{ "pdfBase64": "JVBERi0xLjQKJ..." }`

**方式 B：multipart/form-data**

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `file` | `file` | 是 | 上传的 PDF 文件，仅允许 `application/pdf`。 |

**响应（与方式无关，统一 JSON）**

- **成功**：HTTP 200

| 字段 | 类型 | 说明 |
|------|------|------|
| `ok` | `boolean` | 固定 `true`。 |
| `moduleText` | `string` | AI 提取的模组大纲文本（中文），前端写入当前模组上下文并更新模组预览区。 |

示例：`{ "ok": true, "moduleText": "【黑暗中的低语】\n场景一：老宅……\n关键 NPC：管家……" }`

- **失败**：HTTP 4xx/5xx 或 200 且 `ok: false`

| 字段 | 类型 | 说明 |
|------|------|------|
| `ok` | `boolean` | 固定 `false`。 |
| `error` | `string` | 错误信息（如「未上传文件」「PDF 解析失败」）。 |

**后端建议**：对请求体大小做限制（如 JSON  body ≤ 20MB，multipart 单文件 ≤ 15MB），避免超大 PDF 拖垮服务。

| 项目 | 说明 |
|------|------|
| **方法** | `GET` |
| **路径** | `/api/health` |
| **请求体** | 无 |

**响应**：HTTP 200，JSON

| 字段 | 类型 | 说明 |
|------|------|------|
| `ok` | `boolean` | 固定 `true`。 |
| `message` | `string` | 可选，如 `"Arkham AI 后端运行中"`。 |

示例：`{ "ok": true, "message": "Arkham AI 后端运行中" }`

### 6.4 角色卡保存（已有）

与跑团大厅无直接依赖，供「建立调查员」页同步角色卡到后端使用，保持现有约定即可。

| 项目 | 说明 |
|------|------|
| **方法** | `POST` |
| **路径** | `/api/investigator` |
| **Content-Type** | `application/json` |

| 项目 | 说明 |
|------|------|
| **方法** | `GET` |
| **路径** | `/api/investigator?clientId=xxx` |
| **Query** | `clientId`（必填） |

**响应**：成功 `{ "ok": true, "data": <调查员对象> }`；无数据可返回 404 或 `{ "ok": false, "data": null }`。

### 6.6 接口汇总表

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/api/health` | 健康检查 |
| POST | `/api/keeper/chat` | 守秘人对话（入参：character, moduleContext, chatHistory, userMessage） |

| GET | `/api/investigator?clientId=` | 角色卡获取 |

## 七、小结

| 层级     | 职责简述 |
|----------|----------|
| **前端** | 跑团大厅 UI、用户输入与状态、只调后端 API、展示 API 返回结果 |
| **后端** | 提供 `/api/keeper/chat` 与 `/api/keeper/parse-module`，持有多轮/模组解析逻辑并调用 Gemini，Key 存服务端 |
| **数据流** | 用户输入 → 前端发 HTTP 请求 → 后端 → Gemini → 后端 → 前端展示 |

这样即形成「跑团大厅作为前端与用户交互，用户输入通过 API 发到后端，后端把结果返回前端」的清晰技术架构；后续改代码时可按上述接口与职责拆分实现。
